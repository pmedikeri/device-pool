# Device Enrollment

## Overview

Device Pool supports two enrollment modes for adding devices to the platform:

1. **Bootstrap Token Registration** (preferred) -- A short-lived, one-time-use token generated by the platform, delivered via a bootstrap script that installs and configures the agent automatically.
2. **Username/Password Bootstrap** (fallback) -- The admin provides temporary device credentials through the UI, which the platform uses once to install the agent, then discards immediately.

Both modes result in the same steady-state outcome: the device runs a locally installed agent that authenticates to the platform using a cryptographic device token. The only difference is how the agent gets onto the device in the first place.

---

## Preferred Mode: Bootstrap Token Registration

### Step-by-Step Flow

1. **Admin clicks "Enroll Device" in the web UI.**
   The admin or device owner initiates enrollment from the platform dashboard.

2. **Platform generates a short-lived token.**
   A one-time-use bootstrap token is created with a default TTL of 15 minutes. The token is bound to the user who created it and can only be used once.

3. **UI displays a platform-specific command.**
   The admin copies and runs the command on the target device.

   **Linux / macOS:**
   ```bash
   curl -sSL https://pool.example.com/api/enrollment/bootstrap.sh?token=TOKEN | sudo sh
   ```

   **Windows (PowerShell, run as Administrator):**
   ```powershell
   iwr -useb https://pool.example.com/api/enrollment/bootstrap.ps1?token=TOKEN | iex
   ```

4. **The bootstrap script installs the agent and calls the registration API.**
   See "What the Bootstrap Script Does" below for details.

5. **Platform validates the token, creates the device record, and issues a device credential.**
   The token is consumed (invalidated) on first use. The platform stores a bcrypt hash of the issued device credential.

6. **Agent stores the credential locally and starts its heartbeat loop.**
   From this point forward, the agent authenticates using its device token on every heartbeat and API call.

### What the Bootstrap Script Does

The bootstrap script performs the following steps on the target device:

1. **Creates a dedicated service account.**
   A low-privilege user (e.g., `devicepool-agent`) is created to own the agent process. The agent never runs as root in steady state.

2. **Downloads and installs the agent binary.**
   The correct binary for the device's OS and architecture is fetched from the platform and placed in the appropriate location (`/usr/local/bin/` on Linux, `/usr/local/bin/` on macOS, `C:\Program Files\DevicePool\` on Windows).

3. **Registers with the platform.**
   The script calls the registration API with the bootstrap token. The platform validates the token, creates a device record, and returns a device credential.

4. **Stores the device token securely.**
   The credential is written to a protected configuration file with restrictive permissions (mode `0600`, owned by the service account).

5. **Configures and starts the agent as a system service.**
   - **Linux:** Creates and enables a systemd unit file, then starts the service.
   - **macOS:** Installs a launchd plist and loads it.
   - **Windows:** Registers and starts a Windows service.

### Security Properties

- **One-time use.** Each bootstrap token can only be consumed once. A second attempt with the same token is rejected.
- **Short TTL.** Tokens expire after 15 minutes by default. Expired tokens are rejected even if unused.
- **Tied to creating user.** The token records which admin generated it, providing an audit trail for every enrolled device.
- **No secrets in transit after bootstrap.** The bootstrap token is used exactly once during enrollment. All subsequent communication uses the issued device credential.

## Fallback Mode: Username/Password Bootstrap

### When to Use

- Legacy devices that cannot easily run a bootstrap script.
- Initial migration of an existing fleet onto the platform.
- Devices behind restrictive firewalls where outbound script download is impractical.

### Flow

1. The admin enters the target device's IP address and a set of temporary credentials (username and password) in the enrollment UI.
2. The platform connects to the device over SSH (Linux/macOS) or WinRM (Windows) using those credentials.
3. The platform installs the agent, registers the device, and provisions a device credential -- exactly as the bootstrap script would.
4. The temporary credentials are **discarded immediately** after the bootstrap completes. They are never written to a database, log file, or persistent storage of any kind.

### Credential Handling Guarantees

- **Credentials are NEVER stored in the database.** The platform holds them only for the duration of the bootstrap operation.
- **In-memory only.** If any transient handling is needed (e.g., passing credentials to an internal SSH client), the values are held in encrypted memory and zeroed after use.
- **UI warnings.** The enrollment UI clearly marks this mode as "lower trust" with visible warning labels, encouraging admins to use bootstrap token registration when possible.

### Post-Bootstrap Behavior

Once the agent is installed and registered, the device operates identically to a token-enrolled device. All ongoing authentication uses the issued device token -- the original username and password play no further role.

## Why Passwordless/Token-Based is Preferred

| Concern | Passwords | Device Tokens |
|---|---|---|
| **Replayability** | Easily replayed if intercepted | Single-use bootstrap; long-lived token is never transmitted in replayable form |
| **Guessability** | Subject to brute-force and dictionary attacks | 256-bit random value; computationally infeasible to guess |
| **Sharing** | Often reused across devices or known by multiple people | Unique per device, stored only on that device |
| **Rotation burden** | Requires coordinated password changes | Rotated via re-enrollment; no human involvement needed |
| **Steady-state credential exposure** | Platform must know or be able to derive the password | Platform stores only a bcrypt hash; plaintext exists only on the device |
| **Compromise blast radius** | Leaked password may grant access to other systems | Compromised token is scoped to a single device on this platform |

If the platform database is compromised, an attacker obtains only bcrypt hashes of device tokens -- not passwords that might be reused elsewhere.

## Device Credential Lifecycle

1. **Issuance.**
   At registration time, the platform generates a cryptographically random 256-bit token and returns it to the agent.

2. **Platform-side storage.**
   The platform stores a bcrypt hash of the token in the device record. The plaintext is never persisted on the platform.

3. **Agent-side storage.**
   The agent writes the plaintext token to a protected configuration file:
   - File permissions: `0600` (owner read/write only).
   - File owner: the dedicated service account (e.g., `devicepool-agent`).
   - Location varies by OS (see OS-Specific Notes below).

4. **Usage.**
   The agent includes the token in every heartbeat and API request to authenticate itself to the platform. The platform verifies it against the stored bcrypt hash.

5. **Rotation.**
   Device credentials can be rotated by re-enrolling the device. The old credential is invalidated and a new one is issued. No manual password changes are required.

## OS-Specific Notes

### Linux

| Property | Value |
|---|---|
| **Service manager** | systemd |
| **Service unit** | `/etc/systemd/system/devicepool-agent.service` |
| **Agent binary** | `/usr/local/bin/devicepool-agent` |
| **Config / credential file** | `/etc/devicepool/agent.conf` |
| **Service account** | `devicepool-agent` (nologin shell) |
| **Access method** | SSH |

### macOS

| Property | Value |
|---|---|
| **Service manager** | launchd |
| **Plist** | `/Library/LaunchDaemons/com.devicepool.agent.plist` |
| **Agent binary** | `/usr/local/bin/devicepool-agent` |
| **Config / credential file** | `/etc/devicepool/agent.conf` |
| **Service account** | `devicepool-agent` (dedicated user) |
| **Access method** | SSH |

### Windows

| Property | Value |
|---|---|
| **Service manager** | Windows Service Control Manager |
| **Service name** | `DevicePoolAgent` |
| **Agent binary** | `C:\Program Files\DevicePool\devicepool-agent.exe` |
| **Config / credential file** | `C:\ProgramData\DevicePool\agent.conf` |
| **Service account** | `NT SERVICE\DevicePoolAgent` (virtual service account) |
| **Access method** | RDP |

## Extension Points

The enrollment system is designed to be extended over time:

- **Bulk enrollment via CSV/API.**
  Upload a CSV of device hostnames/IPs or call the enrollment API programmatically to enroll many devices at once. Each device receives its own bootstrap token.

- **Auto-discovery via network scanning.**
  The platform can scan a subnet for discoverable devices (e.g., via mDNS, SNMP, or ARP) and present them as enrollment candidates in the UI.

- **Integration with MDM/SCCM for Windows.**
  Deploy the agent binary and bootstrap token as a software package through Microsoft SCCM, Intune, or other MDM solutions -- no manual script execution required.

- **Certificate-based device identity (future).**
  Replace the shared-secret device token with mutual TLS using per-device X.509 certificates, enabling stronger identity guarantees and compatibility with zero-trust network architectures.
